# 비즈니스 개요

## 문제 정의

대학생의 **78%**가 스터디 그룹을 원하지만, 적합한 팀원을 찾는 데 평균 **2주 이상** 소요된다.

- 에브리타임, 카카오톡 오픈채팅 등 기존 채널은 **정보 비대칭**이 심함
- 실력 수준, 학습 목표, 시간대가 맞지 않아 **중도 이탈률 62%**
- 스터디 운영 도구가 분산되어 있어 **관리 피로도** 높음

## 솔루션: StudyMate

> AI가 학습 성향·수준·목표를 분석하여 최적의 스터디 그룹을 자동 매칭하고, 통합 운영 도구를 제공하는 플랫폼

### 타겟 사용자

1. 전공 시험 대비 스터디를 찾는 대학생
2. 코딩 테스트·자격증 준비 그룹을 원하는 취준생
3. 사이드 프로젝트 팀원을 구하는 개발자

### 수익 모델

| 구분             | 내용                           | 예상 단가   |
| ---------------- | ------------------------------ | ----------- |
| 프리미엄 구독    | AI 심층 매칭, 학습 분석 리포트 | ₩4,900/월   |
| 기업 제휴        | 채용 연계 스터디 스폰서십      | ₩500,000/건 |
| 강의 플랫폼 연동 | 인프런·유데미 제휴 커미션      | 매출의 8%   |

# 핵심 특징 및 차별점

## 1. AI 스마트 매칭 엔진

- **학습 성향 분석**: 온보딩 시 7문항 진단 → MBTI식 학습 유형 분류
- **실력 수준 측정**: 분야별 미니 퀴즈로 객관적 수준 파악
- **호환성 점수**: 시간대, 목표, 성향을 종합한 매칭 스코어 산출
- 매칭 정확도 **89%** (베타 테스트 기준, N=340)

## 2. 통합 스터디룸

- 실시간 화상 미팅 (WebRTC)
- 공유 문서 & 코드 에디터
- 출석 체크 & 진도 트래킹
- 주간 학습 리포트 자동 생성

## 3. 게이미피케이션

- 스터디 연속 출석 스트릭
- 기여도 기반 뱃지 & 랭킹
- 월간 MVP 선정 및 리워드

> "기존 서비스들이 '사람을 찾아주는 것'에 그쳤다면, StudyMate는 '함께 성장하는 환경'까지 제공합니다."

# PRD · 기술 스택 · ERD

## PRD 요약

### 핵심 기능 요구사항

1. **사용자 인증**: 소셜 로그인 (카카오, 구글), JWT 기반 세션 관리
2. **온보딩 진단**: 7문항 학습 성향 설문 → 유형 분류 알고리즘
3. **AI 매칭**: 협업 필터링 + 콘텐츠 기반 하이브리드 추천
4. **스터디룸**: WebRTC 화상, Socket.IO 실시간 채팅, CRDT 공유 문서
5. **대시보드**: 학습 통계 시각화, 주간 리포트 자동 생성
6. **알림 시스템**: FCM 푸시 + 카카오 알림톡

### 비기능 요구사항

- 동시 접속 1,000명 처리 (p99 < 200ms)
- 가용성 99.9% (월간 다운타임 < 43분)
- 개인정보 암호화 (AES-256) 및 GDPR 준수

## 기술 스택

| 계층     | 기술                         | 선택 이유                       |
| -------- | ---------------------------- | ------------------------------- |
| Frontend | React 18 + TypeScript        | 컴포넌트 재사용성, 타입 안정성  |

| AI 서빙  | FastAPI + PyTorch            | 매칭 모델 추론 API              |
| 실시간   | Socket.IO + WebRTC           | 채팅, 화상 미팅                 |
| DB       | PostgreSQL + Redis           | ACID 보장 + 캐시/세션           |
| 인프라   | AWS ECS Fargate + CloudFront | 서버리스 컨테이너, CDN          |

## ERD 주요 엔티티

```
[User] 1──N [StudyMembership] N──1 [StudyGroup]
  │                                      │
  │ 1                                    │ 1
  │ │                                    │ │
  │ N                                    │ N
[LearningProfile]                   [StudySession]
  │                                      │
  │ 1                                    │ 1
  │ │                                    │ │
  │ N                                    │ N
[MatchingScore]                     [Attendance]
```

## 팀원별 회고

### 김도윤 (팀장 / Backend)

프로젝트를 이끌면서 **기술적 의사결정과 일정 관리의 균형**이 가장 어려웠습니다. 매칭 알고리즘을 설계할 때 성능과 정확도 사이의 트레이드오프를 경험했고, 결국 캐싱 전략으로 해결한 것이 가장 뿌듯합니다.

### 이서연 (Frontend)

React Query 도입으로 서버 상태 관리가 훨씬 깔끔해졌습니다. 다만 WebRTC 연동 시 **크로스 브라우저 이슈**가 예상보다 많아 고생했습니다. 다음 프로젝트에서는 초기에 브라우저 호환성 테스트를 먼저 진행하겠습니다.

### 박준혁 (AI / Data)

추천 모델의 **콜드 스타트 문제**가 가장 큰 도전이었습니다. 멘토링 피드백을 반영해 룰 기반 폴백을 추가한 후 초기 사용자 경험이 크게 개선되었습니다. PyTorch → ONNX 변환으로 추론 속도를 3배 향상시킨 것도 의미 있었습니다.

### 최하은 (DevOps)

GitHub Actions + ArgoCD 파이프라인 구축이 처음이라 초반에 삽질을 많이 했습니다. 하지만 한번 구축하고 나니 **배포 자동화의 가치**를 체감했습니다. 모니터링 대시보드 덕분에 베타 테스트 중 장애를 5분 내에 감지할 수 있었습니다.

### 정민서 (기획 / QA)

PRD를 작성하면서 **개발팀과의 소통**이 핵심이라는 것을 배웠습니다. 초기에 기획과 개발 사이의 용어 차이로 혼선이 있었지만, 용어 사전을 만들어 공유한 후 효율이 크게 올랐습니다.

## 느낀점

- **협업의 중요성**: 5명이 각자의 역할에 집중하면서도 매주 동기화한 덕분에 12주 안에 완성도 높은 결과물을 만들 수 있었습니다.
- **멘토링의 가치**: 현업 전문가의 피드백이 프로젝트 방향을 바로잡아 준 결정적 순간이 2번 있었습니다.
- **기술 도전**: 새로운 기술(WebRTC, AI 매칭)을 도입하면서 실패와 학습을 반복했고, 이 과정 자체가 가장 큰 성장이었습니다.
